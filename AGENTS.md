## Multi-agent команда для PageAI

Этот файл описывает виртуальную продуктовую команду (агентов), которая работает над расширением.
Все агенты читают `agent.md` как главный свод правил и архитектурных ограничений.

Перечень разделов настроек (Options и вкладка в панели), ключей `chrome.storage` и пользовательских инструкций агента: **`docs/SETTINGS.md`**.

---

### Расширенные требования расширения

Расширение должно соответствовать следующим требованиям:

#### 1. **Универсальность платформы**

- Работа на любых веб-страницах
- Адаптивность интерфейса под различные типы контента

#### 2. **Управление контекстом**

- Хранение контекста в разрезе страниц (per-page context)
- Сохранение истории чата отдельно для каждой страницы
- Поддержка контекстного окна между сообщениями в пределах одной страницы
- Восстановление контекста при переходе на одну и ту же страницу

#### 3. **Форматирование ответов**

- Обработка и правильный рендер ответов в Markdown формате
- Поддержка списков, заголовков, кода, таблиц, ссылок
- Красивое отображение отформатированного контента в UI
- Поддержка ссылок на источники (в т.ч. текущая страница при саммари)

#### 4. **Текущая страница и MCP**

- **Просмотр текущей страницы**: модель получает контент открытой вкладки и может выдавать по нему краткое саммари по запросу пользователя («о чём эта страница?», «перескажи» и т.п.)
- **MCP в чате**: подключение MCP-серверов в настройках; модель может вызывать инструменты (уведомления, действия) в цикле агента; цепочка размышлений и вызовов сохраняется в ответе
#### 5. **Локальная работа с LLM (BYOM)**

- Интеграция с LM Studio (localhost:1234) для локального выполнения LLM
- Поддержка moq-моделей для быстрого развертывания (текущая: qwen/qwen3-4b-2507)
- Отсутствие зависимостей от облачных API (OpenAI, Gemini и т.д.)
- Возможность переключения между различными локальными моделями

#### 6. **Тестирование с мокируемыми данными**

- Фиксированные тестовые наборы данных для воспроизводимости тестов
- Fake pages для автономного тестирования
- Симуляция различных сценариев (offline, ошибки API, пустые результаты)
- Поддержка записи и воспроизведения HTTP-запросов (VCR-подход)

#### 7. **Безопасность и приватность**

- Локальное хранение всех данных в IndexedDB браузера
- Отсутствие отправки данных на внешние серверы
- Чувствительные данные (API-ключи LLM) хранятся в `chrome.storage.local` (не синхронизируются с аккаунтом)
- Поддержка NDA-требований (no cloud sync, all on-device)

## Цикл разработки: 3 ключевые роли

Расширение разрабатывается в итеративном цикле из трёх основных этапов, каждый с чёткой ответственностью:

---

### 1️⃣ Аналитик (Product & Architecture)

**Цель**: Анализ требований, проектирование решений и планирование работ.

**Ответственность**:

- ✅ Анализ user stories и требований от пользователей
- ✅ Проектирование архитектуры фич в рамках Local-first / Retrieval-first / BYOM
- ✅ Определение Definition of Done для каждой задачи
- ✅ Проверка, что решение не требует backend / облачных API
- ✅ Документирование архитектурных решений в `agent.md`, `README.md`
- ✅ Приоритизирование задач и планирование спринтов

**Входные данные**:

- User stories и feature requests
- Feedback от тестировщика
- Технические ограничения и архитектурные принципы

**Выходные данные**:

- Чётко описанные задачи (user stories + acceptance criteria)
- Архитектурные диаграммы и описания модулей
- DoD и тестовые сценарии
- Сценарии мокирования для разработчика

**Текущие приоритеты**:

- ✅ Chat-интерфейс с историей сообщений (завершено)
- ✅ Persistent хранение истории в IndexedDB (завершено)
- ✅ Просмотр текущей страницы и саммари по запросу (завершено)
- ✅ MCP в чате: вызов инструментов моделью, сохранение цепочки размышлений (завершено)
- 🔄 Интеграция с LM Studio (localhost:1234)
- 🔄 Markdown рендеринг в ChatUI с поддержкой форматирования

---

### 2️⃣ Разработчик (Implementation)

**Цель**: Реализация фич согласно архитектуре и требованиям аналитика.

**Ответственность**:

- ✅ Написание кода на TypeScript с функциональным стилем
- ✅ Разработка модулей согласно спецификации аналитика
- ✅ Минимизация зависимостей (core only)
- ✅ Соблюдение стиля кода и архитектурных принципов
- ✅ Реализация с учётом мокируемых данных (без внешних API)
- ✅ Локальная работа с LM Studio (<http://localhost:1234>)
- ✅ Написание базовых модульных тестов

**Входные данные**:

- Архитектурные описания от аналитика
- Моки данных и сценарии тестирования
- DoD и acceptance criteria

**Выходные данные**:

- Рабочий код в `src/`
- Unit-тесты для ключевых модулей
- Обновление документации по реализации
- Feedback по сложностям архитектуры

**Ключевые модули для разработки**:

- ✅ `src/api/confluence.ts` — REST API клиент Confluence
- ✅ `src/ui/panel.ts` — Chat UI с историей
- ✅ `src/storage/indexdb.ts` — Хранилище чата
- 🔄 `src/llm/client.ts` — Интеграция с LM Studio
- 🔄 `src/search/` — Поиск и markdown рендеринг
- ✅ `src/content/page-extractor.ts` — Парсинг контента текущей страницы для саммари

---

### 3️⃣ Тестировщик (QA)

**Цель**: Обеспечение качества и функциональной корректности расширения.

**Ответственность**:

- ✅ Разработка и выполнение тестовых сценариев
- ✅ Тестирование с мокируемыми данными
- ✅ Проверка интеграции с LM Studio (localhost:1234, модель qwen/qwen3-4b)
- ✅ Валидация доступности и юзабилити
- ✅ Проверка offline-сценариев
- ✅ Поиск regression-дефектов
- ✅ Документирование bug-репортов с воспроизведением

**Входные данные**:

- DoD и acceptance criteria от аналитика
- Реализованный код от разработчика
- Мокируемые тестовые данные

**Выходные данные**:

- Чек-листы тестирования
- Bug-репорты с severity и шагами воспроизведения
- Approval на готовность фичи к production
- Feedback на архитектуру и UX

**Инструменты для тестирования**:

- **LM Studio** (localhost:1234) — локальная модель для быстрого тестирования
- **Mock данные** — фиксированные тестовые данные в `tests/mocks/`
- **Cypress/Playwright** — E2E тесты расширения
- **DevTools** — отладка content scripts и background

**Чек-лист базового функционала**:

- [ ] Расширение загружается без ошибок
- [ ] Side panel открывается на веб-страницах
- [ ] Chat отправляет сообщения и получает ответы (с mock LLM)
- [ ] История чата сохраняется в IndexedDB
- [ ] Поиск работает с мокированными результатами
- [ ] LM Studio интеграция (localhost:1234) функционирует
- [ ] Markdown в ответах рендерится корректно
- [ ] При ошибках загрузки страницы — graceful degradation

### Детальный Workflow для каждой роли

#### 🔍 Аналитик — начало спринта

1. **Получение requirement**
   - Feature request или issue
   - Анализ user story

2. **Проектирование решения**
   - Определение архитектуры (какие модули нужны/изменяются)
   - Проверка на Local-first принципы
   - Определение зависимостей (только core, никаких облачных API)

3. **Подготовка спецификации**
   - DoD (Definition of Done)
   - Acceptance criteria (понятные разработчику)
   - Тестовые сценарии (для тестировщика)

4. **Подготовка mock данных**
   - Какие данные понадобятся для тестов?
   - Какие данные вернут моки?
   - Сохранить в `tests/mocks/`

#### 💻 Разработчик — реализация

1. **Подготовка окружения**

   ```bash
   # Убедитесь, что LM Studio запущен
   # http://localhost:1234
   # Модель: qwen/qwen3-4b-2507
   
   npm install
   npm run dev
   ```

2. **Разработка модулей**
   - Следуйте архитектуре из спецификации
   - Используйте mock данные (import из `tests/mocks/`)
   - Не используйте реальные API запросы (используйте fetch interceptor или mocking library)
   - Тестируйте локально

3. **Интеграция с LM Studio**

   ```typescript
   // Пример
   const response = await fetch('http://localhost:1234/v1/chat/completions', {
     method: 'POST',
     body: JSON.stringify({
       model: 'qwen/qwen3-4b-2507',
       messages: [{ role: 'user', content: 'test' }],
       temperature: 0.7,
     }),
   });
   ```

4. **Unit тесты**
   - Минимум: критические функции
   - Используйте mock data
   - Запускайте локально

#### 🧪 Тестировщик — валидация

1. **Подготовка к тестированию**

   ```bash
   # Убедитесь что всё запущено
   npm run dev
   # LM Studio на localhost:1234
   # Chrome/Edge с загруженным расширением из dist/
   ```

2. **Функциональное тестирование**
   - Следуйте чек-листу из DoD
   - Проверяйте каждый acceptance criterion
   - Используйте mock данные

3. **Проверка LM Studio интеграции**
   - Отправьте запрос в чат
   - Убедитесь что LM Studio отвечает (localhost:1234)
   - Модель qwen/qwen3-4b должна быстро ответить

4. **Bug reporting**
   - Если найдён дефект: создайте issue
   - Укажите: шаги воспроизведения, actual vs expected, скриншот
   - Отправьте обратно разработчику

5. **Approval**
   - Если все критерии пройдены: ✅ Feature READY
   - Иначе: 🔁 Return to Developer

## Интеграция с LM Studio для тестирования

LM Studio развёрнут на **localhost:1234** с моделью **qwen/qwen3-4b-2507**.

### Проверка доступности

```bash
# Проверить, что LM Studio доступна
curl http://localhost:1234/v1/models

# Пример успешного ответа:
# {
#   "object": "list",
#   "data": [
#     {
#       "id": "qwen/qwen3-4b-2507",
#       "object": "model",
#       ...
#     }
#   ]
# }
```

### API использование

```typescript
// Chat completion example
const message = 'Что находится на этой странице?';

const response = await fetch('http://localhost:1234/v1/chat/completions', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    model: 'qwen/qwen3-4b-2507',
    messages: [
      { role: 'system', content: 'Ты помощник пользователя.' },
      { role: 'user', content: message },
    ],
    temperature: 0.7,
    max_tokens: 512,
    stream: false,
  }),
});

const data = await response.json();
console.log(data.choices[0].message.content);
```

### Быстрое тестирование локально

```bash
# В браузере DevTools консоль:
const res = await fetch('http://localhost:1234/v1/chat/completions', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    model: 'qwen/qwen3-4b-2507',
    messages: [{ role: 'user', content: 'Hello!' }],
    max_tokens: 100,
  }),
});
const data = await res.json();
console.log(data.choices[0].message.content);
```
